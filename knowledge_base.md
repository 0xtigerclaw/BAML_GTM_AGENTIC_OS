 - Subnet lifecycle (spawn, pause, retire)

SoraChain’s governance is designed to reflect real value contribution. Power is not distributed to token holders at large, but to **infrastructure operators** who secure, train, and scale the network’s intelligence. This includes:

- **Validator Nodes** – securing the base layer and protocol logic
- **Trainer Nodes** – performing federated learning on local data
- **Aggregator Nodes** – merging contributions into the global model and serving inference; validates trainer node

Each of these roles participates in a **three-layered governance structure**, with **weighted voting based on role, uptime, and stake**.

We do not expect—or desire—retail to play a significant role in governance. Much like Bittensor today, where decisions are led by core validators and foundation infrastructure, **SoraChain prioritizes operator-led stewardship over crowd governance**.

## ✅ **1. Validator Nodes as Audit Agents**
**Document:** Third-party Validation Models 2022643270a980a89994e54f7c0cfe13.md
**Source:** curated_tier1

**Role:** Validators are **not just aggregators** — they act as independent auditors.

- Validators **retrain models locally** on a subset of received updates.
- They **compare output accuracy** on a canonical test set or a rotating validation dataset.
- Results are anchored on-chain for **verifiability and audit**.

📌 *Example:* A validator runs the same model after aggregation on an unseen dataset (e.g., CIFAR-10 test split). If accuracy drops or misaligns with expectations, the trainer's update is flagged.

## 🌐 Subnets (part 1)

- ❓How does Inference work with SoraChain Subnets?
 1. **User submits an inference request** to a specific Subnet.
 2. **Inference (Aggregator) nodes** within that subnet run the model on the input data.
 3. The model’s **provenance (training history)** is verified using on-chain metadata.
 4. The result is returned to the user and optionally **recorded or verified** on-chain.
 
 Detailed explanation: [Inference Pipeline](https://www.notion.so/Inference-Pipeline-1f92643270a9809a86a0c49fb934bbcb?pvs=21) 
 
- ❓How are subnets economically incentivized?
 
 Each subnet has its **own token pool**, fed by staking, enterprise payments, and/or rewards from the main network. Contributors earn based on value-added (training quality, validation, uptime).
 
- ❓Can multiple subnets share data or models?
 
 Yes. Subnets can **export intermediate models** or metadata, optionally registered on-chain. A **universal registry** facilitates cross-subnet collaboration.
 
- ❓What’s the value accrual path for the token via subnets?
 
 Value accrual path for the token via subnets:
 
 - Staking to launch subnets
 - Paying for metadata registration
 - Model inference (B2B and B2C)
 - Inter-subnet bridging (gas fees)
 
- ❓How scalable is the subnet model? Can this support thousands of devices?
 
 
 Yes. Since each subnet manages its own set of trainers/validators asynchronously, the system supports **horizontal scaling**—thousands of edge devices via nested coordination.

1. **Initialization of Process**
 
 The FL pipeline is initiated by setting system parameters, defining the model architecture, and preparing the environment.
 
2. **Model Distribution**
 
 A global model is shared with all participating edge devices or nodes.
 
3. **Local Training**
 
 Each client trains the model on its local dataset without sharing raw data, preserving privacy.
 
4. **Secure Transmission**
 
 The locally trained model updates are encrypted and securely transmitted back to a central server.
 
5. **Secure Aggregation**
 
 The server securely aggregates the updates using privacy-preserving techniques (like differential privacy or homomorphic encryption).
 
6. **Global Updates**
 
 The global model is updated with the aggregated information and redistributed.
 
7. **Iterative Process (Step 3 → 6)**
 
 Steps 3 through 6 are repeated for multiple training rounds to progressively improve the model’s performance.
 
8. **Monitoring & Debugging**
 
 Continuous tracking of system health, model performance, and communication efficiency is conducted throughout the training.
 
9. **Final Deployment**
 
 Once the model meets performance benchmarks, it is deployed for use in production environments.

 - Implemented at the trainer node level before sending updates
